## Problem Statement

Build a hackathon showcase system that uses a Google Coral Dev Board to run real-time
PoseNet inference on an RTSP camera stream, counts exercise reps from keypoint data,
periodically calls a VLM (Vision Language Model) with keypoints + video snapshot for
exercise auto-detection and form coaching, and displays everything in a browser
dashboard — skeleton overlay, rep counter, live coaching text — in near-real-time.

Target exercises: calisthenics only (squats, push-ups, jumping jacks, sit-ups, etc.).
No weight tracking, no multi-person complexity. Just the body.

## What Makes This Cool

- The Coral Edge TPU does real ML inference at 30fps, locally, with no cloud dependency
  for the fast loop. You can point at the hardware and say "that's running a neural net."
- The VLM auto-detects what exercise you're doing — no hardcoded exercise logic. It just
  figures it out from keypoints + image.
- Full pipeline: camera -> edge ML -> rep counting -> VLM coaching -> browser UI. End-to-end.
- Prerecorded video path works identically to live camera, making demos reliable.

## Constraints

- Google Coral Dev Board as the ML inference device (EdgeTPU + PoseNet)
- RTSP stream source (5-10 second inherent latency in live mode)
- Calisthenics exercises only — body keypoints, no objects
- Hackathon showcase format: someone stands in front of a camera, audience watches screen
- Existing repos available: project-posenet, examples-camera, rtsps-viewer

## Premises

1. The Coral Edge TPU runs PoseNet at ~30fps, emitting 17 keypoints per frame. This is
   the "fast brain" — deterministic, real-time, no VLM dependency.
2. Rep counting is deterministic: track a key joint's Y-coordinate over time (e.g., hip
   for squats, wrist for push-ups), detect peaks/valleys. Signal processing, not ML.
3. The 5-10 second RTSP delay makes "live" feel laggy. Pre-recorded video piped through
   the same pipeline is a first-class demo path, not just a fallback.
4. Exercise auto-detection and form coaching are handled by the VLM: send keypoints +
   JPEG snapshot every ~5 seconds. No hardcoded exercise logic needed.
5. The demo UI must be visible on a screen in the room: browser dashboard with skeleton
   overlay on video + rep counter + coaching text feed.

## Approaches Considered

### Approach A: Minimal Single-Process Pipeline
Single Python script on the Coral Dev Board handles everything: RTSP pull, PoseNet,
skeleton overlay, rep counting, VLM calls, Flask UI.
- Effort: M | Risk: Low
- Problem: resource contention between inference, VLM calls, and Flask serving on one process.

### Approach B: Two-Process Edge + Dashboard (CHOSEN)
Process 1 on Coral board: RTSP ingestion + PoseNet inference, publishes keypoints over
a lightweight message channel (ZeroMQ or Redis pub/sub). Process 2 on laptop/server:
subscribes to keypoints, handles rep counting + VLM scheduling, serves browser dashboard
with SVG skeleton overlay composited on HLS video stream.
- Effort: M-L | Risk: Med

### Approach C: All-in-One Container / Laptop Fallback
Docker Compose stack running everything on a laptop, Coral optional via EdgeTPU delegate.
- Effort: L | Risk: Low (demo) but loses the "edge ML on hardware" showcase story.

## Recommended Approach

**Approach B: Two-Process Edge + Dashboard**

### Architecture

```
[RTSP Camera]
     |
     v
[Coral Dev Board -- Process 1: coral-publisher]
  - Pull RTSP stream via OpenCV or GStreamer (from examples-camera/gstreamer patterns)
  - Run PoseNet inference via pose_engine.py (from project-posenet)
  - Emit per-frame messages: { timestamp, keypoints[17], confidence[], jpeg_snapshot }
  - Transport: ZeroMQ PUB socket on tcp://0.0.0.0:5555
  - Also transcode RTSP -> HLS via ffmpeg (from rtsps-viewer) for video display

[Laptop/Server -- Process 2: coaching-server]
  - ZeroMQ SUB socket subscribing to Coral publisher
  - Rep counter: per-exercise joint tracker (hip Y for squats, etc.)
    - Auto-switches exercise when VLM identifies a new exercise type
  - VLM scheduler: every 5 seconds (or on exercise change event), send:
      - Latest JPEG snapshot
      - Last N keypoint frames (as JSON)
      - Prompt: "What exercise is this person doing? How is their form?
                 What should they improve? Count: {rep_count}"
  - WebSocket server (Flask-SocketIO or FastAPI + websockets):
      - Pushes keypoints, rep count, exercise type, coaching text to browser
  - Serves static dashboard HTML

[Browser Dashboard]
  - HLS video playback (from Coral board's ffmpeg HLS output)
  - SVG/Canvas skeleton overlay synced to keypoint stream via WebSocket
  - Rep counter display (large, visible from across the room)
  - Exercise type badge (auto-detected, updates when VLM changes its mind)
  - Coaching text feed (scrolling, most recent VLM output at top)
```

### Key Implementation Notes

**Keypoint publishing (Coral board):**
- Base on `pose_camera.py` + `pose_engine.py` from project-posenet
- Replace the drawing/display logic with ZeroMQ publish
- Publish at **10fps** (downsample from 30fps inference) to reduce network load.
  Rep counting runs in Process 2 at this 10fps rate, which is sufficient for
  calisthenics movements. PoseNet still runs at 30fps internally for inference quality.
- **Snapshot field**: include JPEG snapshot (640x480, base64-encoded) on every 5th
  published frame (every 0.5s). Field is **omitted entirely** (not null) on non-snapshot
  frames. The coaching-server buffers the most recent snapshot for VLM calls.
- **Transport: ZeroMQ msgpack** (not JSON) — JPEG data is binary; msgpack avoids
  base64 encoding overhead over the wire. The message format below uses msgpack types.
- **Coral board CPU risk**: the Coral board (NXP i.MX 8M, 4x Cortex-A53) is not
  powerful enough to run ffmpeg RTSP->HLS transcoding + ZeroMQ publishing simultaneously.
  **Decision**: the laptop/server pulls the RTSP stream directly via ffmpeg for video
  display. The Coral board only handles PoseNet inference + ZeroMQ publishing.
  Step 1 of the build includes a CPU benchmark to validate this assumption.

**Rep counting (coaching-server):**
- Maintain a sliding window of keypoint Y-positions for the rep-relevant joint
- Use scipy.signal.find_peaks or a simple min/max state machine
- Rep threshold configurable per exercise (VLM sets the exercise type, you set
  which joint to track per type)
- **Default exercise on startup**: squat (hip midpoint Y) until first VLM response
  arrives. This avoids an initialization race condition.
- **Exercise change policy**: when VLM returns a different exercise_type than the
  current one, reset the rep counter to 0 and archive the previous set
  (e.g., {"exercise": "squat", "reps": 12}). Log archived sets to console.
- **No pose / low-confidence state**: if all keypoints drop below score 0.5 (person
  out of frame or occluded), freeze the rep counter, hide the skeleton overlay,
  and pause VLM calls. Resume when a confident pose is detected again.
- Suggested joint map:
    squat:        hip midpoint Y
    push-up:      shoulder midpoint Y (or wrist Y if camera is side-on)
    jumping_jack: wrist Y (arms going up/down)
    sit-up:       shoulder Y
    plank:        shoulder Y (should remain near-constant; count time, not reps)
    lunge:        knee Y

**VLM integration:**
- Use OpenAI API (GPT-4o) — best visual reasoning quality; commit to this for the hackathon.
  Local alternatives (Qwen-VL, LLaVA) require a GPU and add infra complexity not worth it here.
- Send JPEG snapshot (from latest buffered snapshot) + last 50 keypoint frames (~5s at 10fps)
  as JSON in the prompt
- Parse response for: exercise_name, form_score (1-10), coaching_tip (1-2 sentences)
- **On VLM JSON parse failure**: retain the last valid VLM response, log the error, and
  retry on the next schedule tick. Never surface a parse error to the UI.
- Rate limit: max 1 call per 5 seconds. Pause calls during "no pose" state.
- Cache last valid response so the UI always shows something even when a VLM call is in-flight.

**Skeleton overlay (browser):**
- PoseNet 17 keypoints: nose, left/right eye, left/right ear, left/right shoulder,
  left/right elbow, left/right wrist, left/right hip, left/right knee, left/right ankle
- Draw connecting lines as Canvas strokes layered above the MJPEG `<img>` element
  (use a transparent `<canvas>` positioned absolutely over the `<img>`)
- Scale keypoint (x,y) — which are 0.0-1.0 normalized — to the `<img>` display dimensions
- Low-confidence keypoints (score < 0.5): draw as dashed/faded or skip
- Hide skeleton entirely during "no pose" state

**Dashboard UI elements:**
- MJPEG video (`<img>`) + Canvas skeleton overlay (absolute positioned, same dimensions)
- Rep counter — large font, visible from across the room
- Exercise type badge — auto-detected, updates when VLM changes its mind
- Form score — numeric display (1-10), color-coded (red < 5, yellow 5-7, green > 7)
- Coaching text feed — most recent VLM tip at top, previous tips below it (log-style)

**Video display (laptop):**
- The laptop runs ffmpeg directly from the RTSP URL to produce an MJPEG stream
  (not HLS). MJPEG has ~1-2s latency vs. HLS's 5-10s, which is better for a
  showcase where someone is standing in front of the camera.
- Serve the MJPEG stream from a simple Flask route: `/video_feed` using multipart
  response. The dashboard `<img>` tag points to this URL.
- This also eliminates the skeleton/video sync problem: 1-2s MJPEG delay is close
  enough to the 0.1s keypoint stream delay that no explicit buffering is needed.
- **ffmpeg MJPEG command** (runs on laptop, not Coral board):
  ```
  ffmpeg -rtsp_transport tcp -i <RTSP_URL> -f mjpeg -q:v 5 pipe:1
  ```

### Message Format (ZeroMQ + msgpack)

Messages are msgpack-encoded dicts. The `snapshot_jpeg` field is **omitted entirely**
on non-snapshot frames (not null — omitting avoids unnecessary dict overhead).

```python
# Every published frame:
{
    "ts": 1712345678.123,      # float, unix timestamp
    "frame_id": 1042,          # int, monotonic counter
    "keypoints": [
        {"name": "nose", "x": 0.52, "y": 0.18, "score": 0.94},
        {"name": "left_shoulder", "x": 0.41, "y": 0.33, "score": 0.91},
        # ... 17 total keypoints
    ],
    # snapshot_jpeg: bytes, present only on every 5th published frame (every 0.5s)
    # field is ABSENT on other frames — check with `"snapshot_jpeg" in msg`
    "snapshot_jpeg": b"<raw jpeg bytes>",
}
```

### VLM Prompt Template

```
You are an AI fitness coach analyzing a workout session.

Current rep count: {rep_count}
Keypoints (last 50 frames = ~5 seconds at 10fps, normalized 0.0-1.0, origin top-left):
{keypoints_json}

Based on the attached JPEG image and keypoints:
1. What exercise is this person performing?
   Choose one: squat, push-up, jumping_jack, sit-up, plank, lunge, or "unknown"
2. Rate their form 1-10 (10 = perfect).
3. Give ONE specific, actionable coaching tip (max 2 sentences).

Respond ONLY in valid JSON with no markdown: {"exercise": "...", "form_score": N, "tip": "..."}
```

## Resolved Decisions

All previously open questions are now resolved:

1. **VLM API**: GPT-4o. Best visual reasoning; worth the cost for a hackathon demo.
2. **Network reliability**: Test with USB ethernet on demo day. WiFi is acceptable for
   ZeroMQ (keypoint data is small). USB ethernet is a backup, not a requirement.
3. **Keypoint/video sync**: Resolved by switching to MJPEG (see below).
4. **HLS vs MJPEG**: **MJPEG**. The laptop runs ffmpeg producing an MJPEG pipe served
   via Flask multipart response. ~1-2s latency, no sync problem, simpler than HLS.
   rtsps-viewer's HLS approach is not used in this project.
5. **Framework**: **FastAPI** (async, cleaner WebSocket support than Flask-SocketIO).
6. **Transport**: **ZeroMQ + msgpack**. Binary-native, fast, no serialization overhead.
